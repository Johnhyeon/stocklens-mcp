# naver-stock-mcp 프로젝트 마스터 문서

> **할루시네이션 없는 AI 주식 분석을 위한 로컬 MCP 서버**
>
> 최종 업데이트: 2026-04-11

---

## 📌 프로젝트 정체성

### 한 줄 설명
Claude에서 자연어로 한국 주식 데이터를 조회하고 분석할 수 있게 해주는 로컬 MCP 서버.

### 핵심 가치
**"AI가 이미지를 추측하는 게 아니라 실제 데이터를 직접 가져와서 분석하게 만든다"**

차트 이미지를 AI에게 보여주면 숫자를 부정확하게 읽어서 할루시네이션이 발생합니다. MCP가 네이버 증권에서 실제 OHLCV/수급/재무 데이터를 가져오면 Claude는 사실 기반으로 정확한 분석을 할 수 있습니다.

### 타겟
- **메인**: 20~50대 직장인 개인 투자자 (비개발자 포함)
- **확장**: AI 활용에 관심 있는 얼리어답터

### 포지셔닝
- **경쟁 우위**: 국내 MCP 기반 한국 주식 분석 시장은 사실상 비어있음 (리서치로 확인)
- **차별점**: 알파스퀘어/씽크풀 같은 앱이 아니라 "AI에게 자연어로 자유롭게 질문"할 수 있는 구조
- **데이터 소스**: 네이버 증권 공개 데이터 (API 키 불필요)

## 🏗 아키텍처 원칙

### "MCP는 데이터 공급자, Claude는 요리사"

```
MCP = 네이버의 분류/랭킹/리스트를 그대로 가져오는 재료 공급자
Claude = 재료를 조합해서 사용자 질문에 맞는 답을 만드는 요리사
사용자 = 어떤 재료에서 출발할지, 어떤 결론을 원하는지 명시적으로 질문
```

### 왜 이 설계인가
1. **유연성 무한대**: 새로운 조건부 질문마다 도구를 추가할 필요 없음
2. **영상 컨텐츠 차별화**: "프롬프팅 기술"이 핵심 가치가 됨
3. **사용자 학습**: 질문을 잘하는 법을 익히면 같은 도구로 완전히 다른 결과 도출

### 하면 안 되는 것
- 스크리닝 알고리즘을 MCP 코드에 박는 것 (함수 무한 증식)
- "PER 10 이하 종목 찾기" 같은 조건부 함수 만들기 (Claude가 할 일)

## 🚀 v2 상세 설계

### 추가할 도구 7개

#### 테마/업종 계열 (4개)
```
list_themes()                  → 네이버 테마 목록 (80개 내외)
get_theme_stocks(theme_name)   → 특정 테마 종목 리스트
list_sectors()                 → 업종 목록
get_sector_stocks(sector_name) → 특정 업종 종목 리스트
```

#### 랭킹 계열 (3개)
```
get_volume_ranking(market, count)     → 거래량 상위
get_change_ranking(direction, count)  → 등락률 상위/하위
get_market_cap_ranking(market, count) → 시가총액 상위
```

### 데이터 소스
- 테마: `finance.naver.com/sise/theme.naver`
- 테마 상세: `finance.naver.com/sise/sise_group_detail.naver?no=XXX`
- 업종: `finance.naver.com/sise/sise_group.naver?type=upjong`
- 거래량: `finance.naver.com/sise/sise_quant.naver`
- 등락률: `finance.naver.com/sise/sise_rise.naver` / `sise_fall.naver`
- 시가총액: `finance.naver.com/sise/sise_market_sum.naver`

### 응답 크기 제어

| 도구 | 기본값 | 최대값 | 비고 |
|------|--------|--------|------|
| `list_themes` | 50 | 80 | 네이버에 전체 80개 내외 |
| `get_theme_stocks` | 전체 | 100 | 테마당 평균 10~50개 |
| `get_volume_ranking` | 50 | 100 | Claude 컨텍스트 한도 고려 |
| `get_change_ranking` | 50 | 100 | 동일 |
| `get_market_cap_ranking` | 50 | 100 | 동일 |

### v2로 가능해지는 질문 예시

## 💡 프롬프팅 레벨링

### Level 1 — 기초 (무료 영상 1편에서 전달)
- 왜 할루시네이션이 생기는가
- 데이터를 직접 넣어주면 해결
- 좋은 질문의 3요소:
  - **데이터 범위** (예: "60일 일봉", "20일 수급")
  - **분석 기준** (예: "이동평균선 기준", "RSI로")
  - **원하는 결론** (예: "매수/매도 판단", "표로 정리")

### Level 2 — 중급 (무료 영상 2편에서 전달)
- MCP에 스크리닝 도구 추가
- "진입점"의 개념: 어디서부터 검색할지 명시해야 함
- 조건부 필터링 프롬프트 작성법

### Level 3 — 고급 (유료 강의)
- **모듈 1**: 시장 전체 읽기 (코스피/코스닥 지수, 업종별 자금 이동, 테마 순환)
- **모듈 2**: 진입점 선정 (거래량 급증, 외국인 순매수, 52주 신고가 등)
- **모듈 3**: 종목 정밀 분석 (차트+수급+재무 크로스체크)
- **모듈 4**: 나만의 투자 루틴 만들기
- **모듈 5**: AI 활용 심화 (프롬프트 체이닝, 컨텍스트 관리, 결과 검증)

## 💰 수익화 원칙

### MCP 자체를 팔지 않는다
MCP는 프로토콜이고 네이버 데이터는 공개다. 무료 유지가 전략적으로 맞음.

### 돈 내는 건 결국 "나 혼자 못 하는 것"
- **도구 사용법**은 안 팔림 (README만 봐도 됨)
- **체계적인 프레임워크**는 팔림 (고급 강의)
- **결과물**은 팔림 (1:1 세팅 서비스)

### 빌드업 원리
```
같은 시스템을 난이도별로 3번 판매:
  1. 도구 (MCP) → 직접 하고 싶은 사람 (무료)
  2. 지식 (강의) → 만들고 싶은 사람 (유료 1회)
  3. 결과 (서비스) → 귀찮은 사람 (유료 서비스)
```

## 🗂 개발 브랜치 전략

### 원칙
**새로운 기능이나 큰 변경 작업은 반드시 별도 브랜치에서 진행.** 기능 완성 후 상위 브랜치로 merge.

### 브랜치 계층 구조
```
main                              ← 배포된 안정 버전 (현재 v0.1.2)
 └─ develop/v2                    ← v2 메이저 기능 통합 브랜치
     └─ feature/v2-performance    ← 성능 최적화 (캐시, 싱글톤 등)
     └─ feature/v2-xxx            ← 추후 다른 v2 기능 작업 시
```

### 네이밍 컨벤션
- `develop/v<숫자>` — 메이저 버전 개발 통합 브랜치
- `feature/v<숫자>-<기능명>` — 특정 기능 작업 브랜치
- `fix/<버그명>` — 핫픽스 (main에서 직접 분기)

### 업데이트 배포 흐름
1. 기능 브랜치에서 개발 (예: `feature/v2-performance`)
2. 완료 후 `develop/v2`로 merge
3. v2 전체 테스트 후 `main`으로 merge
4. `pyproject.toml` version bump
5. `python -m build && python -m twine upload dist/*`
6. 구글폼 수집 이메일에 업데이트 공지 발송

### 사용자 업데이트 방법
- **자동 업데이트 없음** (Python 패키지 특성)
- `update.bat` 또는 `pip install --upgrade naver-stock-mcp` 수동 실행

이 파일은 **프로젝트의 단일 진실 출처(Single Source of Truth)**입니다.

새로운 결정이나 변경사항이 생길 때마다 여기 업데이트하고, Notion에도 동기화합니다. 새 대화 세션을 시작할 때 이 파일을 읽히면 전체 맥락을 즉시 파악할 수 있습니다.
